那些年我们在商城奋斗的光辉岁月 - redis超时(延迟高)导致的事故

发表于 2021-02-28 更新于 2023-02-18 分类于 stability 阅读次数： 阅读次数： Valine：
本文字数： 16k 阅读时长 ≈ 15 分钟

2月26，元宵节当天，商城出了一个事故，😭，简单地说是一个 小bug + redission使用不当 + 高并发流量 + 强校验 导致的一个大问题。

(1) 问题时间线

02-25 20:48 新需求上预发验证时，遇到个小问题，代码里忘记判空，导致某个key缓存了全表数据(全量活动标签)，随后redis集群里的一个分片 CPU飙高开始报警；
02-25 21:00 预发代码回滚完成。
　　但是缓存没有清理。而且线上和预发用的是同一个redis集群。(给事故埋下伏笔)
　　由于晚上流量逐渐变小，所以线上未出现问题。

02-26 08:32 活动服务的redis查询延时升高告警
02-26 08:35 ～ 09:02 redis连接延时升高、导致查询活动信息延时升高，下单调活动超时，触发报警 (事故开始)

02-26 09:02 由于天降惊喜活动相关查询减少，慢请求减少，redis QPS下降，redis proxy恢复正常，活动接口耗时下降，下单服务调活动接口恢复正常 (事故结束)
02-26 13:05 删除问题缓存，代码修复发布上线，redis查询QPS趋于稳定，完全恢复

8:48 ~ 8:55是查询最慢的时间 监控


(2) 问题分析

(2.1) 事故直接原因

  1. redis某个分片查询耗时高(最高有200ms)
  2. redis proxy查询耗时高(300-500ms)，导致活动接口耗时高
  3. 活动接口耗时高(>500ms) 导致 下单调活动超时
  4. 下单强依赖活动，调活动失败后返回网络错误

(2.2) 问题原因

请求路径 redis client –> VIP:PORT –> 均摊到16个proxy机器 –> 热点key+大key落到的某一台redis分片

由于请求在proxy堆积，导致所有的proxy的cpu都飙升了。

备注 redis查询耗时 2~200ms，而redis proxy耗时有 300 ~ 500 ms

(2.2.1) 为什么redis有一个分片查询耗时高？

以前redis所有分片的查询都在1~5ms，而事故发生时redis有1个分片耗时最高有200ms，为什么？

  1. 事故发生前一个有个bug，导致出问题的redis分配里某个key是大key。 (key对应的value是全表的数据)
  2. redis出问题那个分片上的查询是个复杂lua脚本，会操作hash zset，而且会对hash里的所有field操作，特别耗费性能。
  3. 代码里有循环调redis的情况，正好查的是那个分片
  4. 调redis超时后redission默认重试3次，导致流量翻倍
  5. 出问题那个分片上的流量还比较大，相当于热key

大key + 热key + 复杂lua脚本 + 流量翻倍 导致有1个redis分片耗时高(200ms)

(2.2.2) 为什么所有redis proxy查询耗时高?

以前redis proxy查询<1ms，而事故发生时16个redis proxy耗时都有 300 ~ 500 ms ，为什么?

  1. (根因) Redis有1个分片的 热key + 大key + 复杂lua脚本 导致redis的请求耗时高。
  2. (直接原因) 请求Redis耗时高，并且流量大，而请求均匀分布在16个redis proxy，请求堆积，导致所有redis proxy耗时高。
  3. 调用redis失败后又新建连接，导致redis proxy连接数增加，性能下降，延迟增加。
  4. 代码里有循环调redis的情况，导致redis proxy CPU压力较大
  5. 调redis-proxy超时后redission默认重试3次，导致流量翻倍 (1~3倍)
  6. 用户有重试，导致流量翻倍 (2~10倍)

redis一个分片耗时高 + 请求堆积 + 连接数增多 + 流量翻倍 综合导致redis proxy CPU飙升，请求堆积，所有请求都耗时。

proxy是请求堆积才引起的CPU利用率升高，
根因还是有一个Redis分片上请求变慢，Redis变慢的原因是这个分片上有lua脚本访问热key，请求量比别的分片高

相同qps，连接数增加，延迟也会大幅增加


(2.2.3) 为什么活动查询接口耗时高？

以前活动接口查询耗时在30~200ms，为什么事故发生时耗时超过500ms ?

  1. 调redis获取数据耗时高，有300 ~ 500ms。
  2. 代码里有循环调redis的情况，导致耗时翻倍。
  3. 活动查询标签的代码里有默认查询新用户的逻辑(这个场景不应该走这段代码)，也比较耗时。
  4. 查询活动接口会判断这个商品是否是秒杀、新人、万人团、促销，一共会调用4次redis，耗时较高
  5. 活动服务里redis的默认超时时间是3秒
  6. 调redis没有熔断机制。
  7. 调redis超时后，又创建tcp连接请求redis，共有 64个tcp连接/台 * 200台机器 = 12800个tcp连接
  8. 活动服务web容器(Tomcat)连接数被打满。
  9. 调用方流量变大(用户重试)，导致流量放大。

备注：活动服务redis配置了每台机器最多64个连接

(2.2.4) 为什么下单失败了？

为什么以前交易下单调活动没有失败？

  1. 交易系统-下单服务 调 活动超时后报错。
  2. 下单对活动是强依赖 (历史原因 下单要拿到活动信息给秒杀商品超卖兜底)。
  3. 用户重试 流量翻倍，导致活动接口失败数更多。

(2.3) 问题lua脚本

Redis提供的hash只有key有过期时间，而hash对象的 field value上是没有过期时间的。
这个lua脚本的作用是给 hash结构里的 field value 设置一个过期时间。

下面命令比较复杂，简化一下就是 EVAL lua_script 5 key1 key2 key3 key4 key5 value1 value2
script是第一个参数，为Lua脚本。
5代表后续参数有5个key。
key1 key2 key3 key4 key5，是要操作的键。
value1 value2，代表有2个参数，在lua脚本中通过ARGV[1], ARGV[2]获取。

下面是根据redis slowlog拿到的一个样例

key1  "pc:goods_label:17"
key2  "redisson__timeout__set:{pc:goods_label:17}"
key3  "redisson__idle__set:{pc:goods_label:17}"
key4  "redisson__map_cache__last_access__set:{pc:goods_label:17}"
key5  "{pc:goods_label:17}:redisson_options"
value1  "1614301199559"
value2  "[\"java.lang.Long\",1250]", reply raw: ":1\r\n"

lua脚本内容，注释是为了方便理解加的

-- KEYS[1] 是入参里的key1 对应hash对象的key  "pc:goods_label:17"
-- KEYS[2] 是入参里的key2 对应zscore的key   "redisson__timeout__set:{pc:goods_label:17}"
-- KEYS[3] 是入参里的key3 对应zscore的key   "redisson__idle__set:{pc:goods_label:17}"
-- KEYS[4] 是入参里的key4 对应              "redisson__map_cache__last_access__set:{pc:goods_label:17}"
-- KEYS[5] 是入参里的key5 对应              "{pc:goods_label:17}:redisson_options"
-- ARGV[1] 是入参里的value1 对应 过期时间    "1614301199559"
-- ARGV[2] 是入参里的value2 对应 hash对象的field 也是zadd的member   "[\"java.lang.Long\",1250]", reply raw: ":1\r\n"
-- 
-- 使用hash对象获取 key field对应的值
local value = redis.call('hget', KEYS[1], ARGV[2]);
-- hget key field 对应的值为空
if value == false then
    return nil;
end ;

-- 将二进制反序列化，解析数据
local t, val = struct.unpack('dLc0', value);
-- 默认过期时间 无限大
local expireDate = 92233720368547758;
-- 从zset里获取field对应的过期时间   zscore key member
local expireDateScore = redis.call('zscore', KEYS[2], ARGV[2]);
-- 过期时间 不等于 false
if expireDateScore ~= false then
    -- 解析过期时间
    expireDate = tonumber(expireDateScore)
end ;

-- t不等于0
if t ~= 0 then
    local expireIdle = redis.call('zscore', KEYS[3], ARGV[2]);
    -- expireIdle 不等于 false
    if expireIdle ~= false then
        if tonumber(expireIdle) > tonumber(ARGV[1]) then
            -- 将二进制反序列化，
            local value = struct.pack('dLc0', t, string.len(val), val);
            -- 更新 hash对象 key field对应的值
            redis.call('hset', KEYS[1], ARGV[2], value);
            -- 更新zset对象 key 的值
            redis.call('zadd', KEYS[3], t + tonumber(ARGV[1]), ARGV[2]);
        end ;
        -- 过期时间
        expireDate = math.min(expireDate, tonumber(expireIdle))
    end ;
end ;

-- 如果 过期时间 <= 入参里的value1过期时间
if expireDate <= tonumber(ARGV[1]) then
    return nil;
end ;

-- hget key5 'max-size'
local maxSize = tonumber(redis.call('hget', KEYS[5], 'max-size'));
-- 如果 max-size 对饮的值为空 并且 不等于0
if maxSize ~= nil and maxSize ~= 0 then
    -- zadd key4 score(对应入参value1的时间) member(对应hash的field)
    redis.call('zadd', KEYS[4], tonumber(ARGV[1]), ARGV[2]);
end ;

-- 返回值
return val;

lua脚本给hash key field设置过期时间原理

从上面的lua脚本可以看出 redisson给hash对象每个field设置过期时间的原理

  1. 对hash对象的key和field 对应设置过期时间
    zadd key score(过期时间) member(对应hash对象的field)

  2. 在hget key field时获取过期时间
    通过 zscore key member(对应hash对象的field) 获取分数值(score)，这个score就是过期时间

下面是用redis命令实现上面lua脚本的功能
hset key1 field1 value1
zadd key4(≈key1) score(过期时间) member(对应hset里的field1)
zscore key3(≈key1) member(对应haset里的field1) 获取的值就是过期时间

key4和key1是关联的， key4 = "redisson__map_cache__last_access__set:{" + key1 + "}"
key3和key1是关联的， key3 = "redisson__idle__set:{ + key1 + }"


对应到这次的数据就是
hset "pc:goods_label:17" "1250" "List<ActivityGoodsLabel>数据"
zadd "redisson__map_cache__last_access__set:{pc:goods_label:17}" "1614301199559" "1250"
zsocre "redisson__idle__set:{pc:goods_label:17}" "1250" 获取分数值(score) 这个score就是过期时间

(2.4) 问题代码

活动接口 /getActivityGoodsList 获取活动信息时需要获取活动商品列表，其中需要商品活动标签信息

/**
 * 获取商品活动标签信息
 * 
 * 最终拿到的数据是 Map<goodId, List<ActivityGoodsLabel>>   
 * goodId 是 Long类型
 * ActivityGoodsLabel 是 活动标签对象
 */
@Override
public Map<Long, List<ActivityGoodsLabel>> getActiveLabelByGoodsIds(Long cityId, List<Long> goodsIdList, List<Integer> typeList) {
    if(CollectionUtils.isEmpty(goodsIdList)){
        return Collections.emptyMap();
    }

    try {
        // key是 goods_label:cityId
        RMapCache<Long, List<ActivityGoodsLabel>> mapCache = redissonClient.getMapCache(buildCacheKey(cityId));

        List<Long> needQueryList = Lists.newArrayList();
        Map<Long, List<ActivityGoodsLabel>> resultGoodsLabelMap = Maps.newHashMap();

        if (MapUtils.isNotEmpty(mapCache)) {
            // 循环里调mapCache.get() 踩坑了 
            goodsIdList.forEach(goodsId -> {  // 按照goodsId逐个设置label
                if (mapCache.containsKey(goodsId)) {
                    // 缓存查到部分数据  // 这块mapCache.get(goodsId) 相当于hget goodsId 
                    resultGoodsLabelMap.put(goodsId, mapCache.get(goodsId));
                } else {
                    //未查到的数据
                    needQueryList.add(goodsId);
                }
            });

            if (CollectionUtils.isEmpty(needQueryList)) {
                return resultGoodsLabelMap;
            }
        }else {
            // 需要从数据库查的goodsIdList
            needQueryList.addAll(goodsIdList);
        }
        // 从数据库查询goodsIds对应的标签
        Map<Long, List<ActivityGoodsLabel>> goodsLabelMap = doQueryLabel(typeList, needQueryList);

        // 把从数据库查到的数据添加到缓存
        // 循环调用redis 
        goodsLabelMap.forEach((key, val) -> mapCache.put(key, val, 5, TimeUnit.SECONDS));

        resultGoodsLabelMap.putAll(goodsLabelMap);
        //返回值设置
        return resultGoodsLabelMap;

    }catch (Exception e){
        log.warn("查询label出现异常，err:{}", e.getMessage(), e);
        return doQueryLabel(typeList, goodsIdList);
    }
}

从数据库根据商品id查询活动标签的方法

private Map<Long, List<ActivityGoodsLabel>> doQueryLabel(List<Integer> typeList, List<Long> needQueryList) {
    // 新加的判空    大事故前的小bug
    if(CollectionUtils.isEmpty(needQueryList)){
        return Collections.emptyMap();
    }
    
    LambdaQueryWrapper<ActivityGoodsLabel> queryWrapper = Wrappers.lambdaQuery();
    long currentTime = TimeUtils.getCurrentTime();
    queryWrapper.in(ActivityGoodsLabel::getGoodsId, needQueryList)
            .eq(ActivityGoodsLabel::getStatus, 0)
            .in(ActivityGoodsLabel::getActivityType, typeList)
            .le(ActivityGoodsLabel::getStartTime, currentTime)
            .ge(ActivityGoodsLabel::getEndTime, currentTime)
    .select(ActivityGoodsLabel::getGoodsId,ActivityGoodsLabel::getActivityType);

    // 查数据库
    List<ActivityGoodsLabel> labels = activityGoodsLabelMapper.selectList(queryWrapper);

    // 按照goodsId分组
    return labels.stream().collect(Collectors.groupingBy(ActivityGoodsLabel::getGoodsId));
}

(3.2) 为什么所有proxy的cpu都飙高？

lua只是跑在一个redis分片上

  1. 根因是有一个Redis分片上请求变慢，Redis变慢的原因是这个分片上有lua脚本访问热key+大key
  2. redis proxy响应变慢，导致活动服务启动了更多的连接，直到达到活动集群连接池上限64*200=12800，那么每个proxy的连接数很高，导致proxy的CPU飙升；

(3.3) 为什么所有redis proxy都有阻塞？

  1. 所有redis proxy的CPU利用率升高后，一定也会影响proxy处理请求的耗时。
  2. Redis的一个分配有热key+大key，每个redis proxy上均匀分布着慢请求，同时由于在proxy的每个连接的请求是有序的，所以当慢请求未响应时，后面的请求也无法响应。
  3. 当时redis的慢请求响应耗时99%分位在200+ms以内，那么正常的请求也会由于proxy在等待慢请求响应而超时。

1.redis慢查询出现的原因？
redission中的Map使用了复杂的lua脚本和redis进行交互，lua脚本很耗性能，这些key全部落在一个分片上，当请求量上升时，对这个分片的查询响应变慢。

2.为什么慢查询会阻塞正常的请求？
a. redis 响应变慢，导致活动服务启动了更多的连接，直到达到集群连接池上限64*200=12800，那么每个proxy的连接数很高，导致proxy的CPU满载；
b. 每个proxy上均匀分布着慢请求，同时由于在proxy的每个连接的请求是有序的，所以当慢请求未响应时，后面的请求也无法响应。当时redis的慢请求响应耗时99%分位在200+ms以内，那么正常的请求也会由于proxy在等待慢请求响应而超时。
c. 活动服务对于redis的连接配置按照默认设置，超时时间默认3秒，默认重试3次，接口无法快速失败

3.正常请求超时后，为什么下单会失败？单量下降了66%？
redis的默认超时时间是3秒，查询活动接口会判断这个商品是否是秒杀、新人、万人团、促销，一共会调用redis 4次只要有一个请求超时未返回，就会导致上游接口超时

(5.3.2) redisson配置优化

package com.xiaoju.wujie.price.infrastructure.config;

import org.redisson.Redisson;
import org.redisson.config.Config;
import org.springframework.beans.factory.annotation.Value;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.scheduling.concurrent.ThreadPoolTaskExecutor;

import java.util.concurrent.ThreadPoolExecutor;

/**
 * 
 */
@Configuration
public class RedissonConfig {

    @Value("${redis.url}")
    private String redisUrl;

    @Value("${redis.poolsize}")
    private int poolSize;

    @Bean(value = "redisson")
    public Redisson getRedisson() {
        Config config = new Config();
        Redisson redisson = null;
        try {
            config.useSingleServer()
                    .setAddress(new StringBuilder().append("redis://").append(redisUrl).toString())
                    .setFailedAttempts(0) // 设置失败重试次数
                    .setConnectTimeout(100) // 设置链接超时时间
                    .setRetryAttempts(1) // 设置重试次数 
                    .setTimeout(100); // 设置超时时间

            redisson = (Redisson) Redisson.create(config);
            return redisson;
        } catch (Exception e) {
            log.error("getRedisson error {}", e);
        }
    }
    @Bean
    public ThreadPoolTaskExecutor redisCacheExecutor() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        // 设置核心线程数
        executor.setCorePoolSize(2);
        // 设置最大线程数
        executor.setMaxPoolSize(4);
        // 设置队列容量
        executor.setQueueCapacity(100);
        // 设置线程活跃时间（秒）
        executor.setKeepAliveSeconds(200);
        // 设置默认线程名称
        executor.setThreadNamePrefix("RedissonPlus-");
        // 设置拒绝策略
        executor.setRejectedExecutionHandler(new ThreadPoolExecutor.AbortPolicy());
        // 等待所有任务结束后再关闭线程池
        executor.setWaitForTasksToCompleteOnShutdown(true);
        return executor;
    }

}
