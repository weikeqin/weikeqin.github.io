redisson使用不当造成的重大事故

发表于 2021-02-28 更新于 2022-12-25 分类于 java 阅读次数： 本文字数： 11k 阅读时长 ≈ 10 分钟

2月26，元宵节当天，出了一个P1事故，:hurt:，简单地说是一个小bug导致查库后的数据放入某个分配的缓存里，缓存然后没清理，然后第二天秒杀时线上代码使用了这个缓存，再叠加逻辑不合理+redis配置不合理，导致下单时调活动超时，下单时活动校验未通过，影响单量数万单。

(1) 问题时间线

02-25 20:48 业务的一个新需求上预发，但是在预发验证时，发现代码里忘记判空，导致验证的商品缓存了全量的活动标签，随后redis集群里的一个分片 CPU飙高开始报警；
02-25 21:00 预发代码回滚完成。
但是缓存没有清理。而且线上和预发用的是同一个redis集群。
由于晚上流量逐渐变小，所以线上未出现问题。
02-26 08:32 活动服务的redis查询延时升高告警
02-26 08:35 ～ 09:02 redis连接延时升高、导致查询活动信息延时升高，下单下跌报警触发
02-26 08:55 price-center 活动查询接口超时，原因是10个proxy连接都被打满, CPU使用率接近100%，导致后面的连接响应超时，同时接口响应超时
02-26 09:02 由于天降惊喜相关查询减少，慢请求消失，redis QPS下降，下单恢复正常
02-26 13:05 删除问题缓存，代码修复发布上线，redis查询QPS趋于稳定，完全恢复

10个proxy，单个1500连接左右，redis proxy共支持15000的连接
活动单台机器默认64个连接，共200台机器，共支持12800的连接

事故发生期间，redis慢查询的请求量翻倍(用户重试+redission默认重试3次)，由于慢查询耗时长响应慢，所以在同一个连接上的之后的请求响应也会变慢，导致了更多的请求响应变慢；

慢查询是由于redission的Map结构在put时使用的是lua脚本形式。

8:48 ~ 8:55是查询最慢的时间 监控

(2.1) 问题代码

活动接口 /getActivityGoodsList 获取活动信息时需要获取活动商品列表，其中需要商品活动标签信息

/**
 * 获取商品活动标签信息
 */
@Override
public Map<Long, List<ActivityGoodsLabel>> getActiveLabelByGoodsIds(Long cityId, List<Long> goodsIdList, List<Integer> typeList) {
    if(CollectionUtils.isEmpty(goodsIdList)){
        return Collections.emptyMap();
    }

    try {
        // key是 goods_label:cityId
        RMapCache<Long, List<ActivityGoodsLabel>> mapCache = redissonClient.getMapCache(buildCacheKey(cityId));

        List<Long> needQueryList = Lists.newArrayList();
        Map<Long, List<ActivityGoodsLabel>> resultGoodsLabelMap = Maps.newHashMap();

        if (MapUtils.isNotEmpty(mapCache)) {
            // 按照goodsId逐个设置label
            goodsIdList.forEach(goodsId -> {
                if (mapCache.containsKey(goodsId)) {
                    //缓存查到部分数据
                    resultGoodsLabelMap.put(goodsId, mapCache.get(goodsId));
                } else {
                    //未查到的数据
                    needQueryList.add(goodsId);
                }
            });

            if (CollectionUtils.isEmpty(needQueryList)) {
                return resultGoodsLabelMap;
            }
        }else {
            // 需要从数据库查的goodsIdList
            needQueryList.addAll(goodsIdList);
        }
        // 从数据库查询goodsIds对应的标签
        Map<Long, List<ActivityGoodsLabel>> goodsLabelMap = doQueryLabel(typeList, needQueryList);

        // 把从数据库查到的数据添加到缓存
        goodsLabelMap.forEach((key, val) -> mapCache.put(key, val, 5, TimeUnit.SECONDS));
        resultGoodsLabelMap.putAll(goodsLabelMap);
        //返回值设置
        return resultGoodsLabelMap;

    }catch (Exception e){
        log.warn("查询label出现异常，err:{}", e.getMessage(), e);
        return doQueryLabel(typeList, goodsIdList);
    }
}

private Map<Long, List<ActivityGoodsLabel>> doQueryLabel(List<Integer> typeList, List<Long> needQueryList) {
    // 判空 <!--2021-01-28 新加的判空 -->
    if(CollectionUtils.isEmpty(needQueryList)){
        return Collections.emptyMap();
    }
    // 查数据库
    LambdaQueryWrapper<ActivityGoodsLabel> queryWrapper = Wrappers.lambdaQuery();
    long currentTime = TimeUtils.getCurrentTime();
    queryWrapper.in(ActivityGoodsLabel::getGoodsId, needQueryList)
            .eq(ActivityGoodsLabel::getStatus, 0)
            .in(ActivityGoodsLabel::getActivityType, typeList)
            .le(ActivityGoodsLabel::getStartTime, currentTime)
            .ge(ActivityGoodsLabel::getEndTime, currentTime)
    .select(ActivityGoodsLabel::getGoodsId,ActivityGoodsLabel::getActivityType);

    List<ActivityGoodsLabel> labels = activityGoodsLabelMapper.selectList(queryWrapper);

    return labels.stream().collect(Collectors.groupingBy(ActivityGoodsLabel::getGoodsId));
}

public <K, V> RMapCache<K, V> getMapCache(String name) {
    return new RedissonMapCache<K, V>(evictionScheduler, connectionManager.getCommandExecutor(), name, this, null);
}
 
 
public RedissonMapCache(EvictionScheduler evictionScheduler, CommandAsyncExecutor commandExecutor,
                        String name, RedissonClient redisson, MapOptions<K, V> options) {
    super(commandExecutor, name, redisson, options);
    if (evictionScheduler != null) {
        evictionScheduler.schedule(getName(), getTimeoutSetName(), getIdleSetName(), getExpiredChannelName(), getLastAccessTimeSetName());
    }
}

Redisson通过在初始化RedissonMapCache时，设置了一个EvictionScheduler,这个类通过netty的EventloopGroup线程池周期地向以redisson_map_cache_expired前缀名的频道发布消息，RedissonMapCache会订阅这个频道来处理消息。

(2.3) 问题原因

redis的请求路径 redis client–> VIP:PORT –> 均摊到16个proxy机器 –> 由于热点key落到的某一台redis机器
由于请求在proxy堆积，导致所有的proxy的cpu都飙升了。

redis大量慢查询导致正常查询阻塞，交易系统-下单服务调用活动接口超时，导致下单失败；

1.redis慢查询出现的原因？
redission中的Map使用了复杂的lua脚本和redis进行交互，lua脚本很耗性能，这些key全部落在一个分片上，当请求量上升时，对这个分片的查询响应变慢。

2.为什么慢查询会阻塞正常的请求？
a. redis 响应变慢，导致活动服务启动了更多的连接，直到达到集群连接池上限64*200=12800，那么每个proxy的连接数很高，导致proxy的CPU满载；
b. 每个proxy上均匀分布着慢请求，同时由于在proxy的每个连接的请求是有序的，所以当慢请求未响应时，后面的请求也无法响应。当时redis的慢请求响应耗时99%分位在200+ms以内，那么正常的请求也会由于proxy在等待慢请求响应而超时。
c. 活动服务对于redis的连接配置按照默认设置，超时时间默认3秒，默认重试3次，接口无法快速失败

3.正常请求超时后，为什么下单会失败？单量下降了66%？
redis的默认超时时间是3秒，查询活动接口会判断这个商品是否是秒杀、新人、万人团、促销，一共会调用redis 4次只要有一个请求超时未返回，就会导致上游接口超时

问题总结：
活动接口限流降级方案不够完善
redis对故障分片预案不够完善：redis集群某个分片出现问题，导致集群所有proxy的连接请求升高，CPU升高
下单对于活动查询是强依赖，无法降级
修改redis的超时时间

优化

package com.xiaoju.wujie.price.infrastructure.config;

import org.redisson.Redisson;
import org.redisson.config.Config;
import org.springframework.beans.factory.annotation.Value;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.scheduling.concurrent.ThreadPoolTaskExecutor;

import java.util.concurrent.ThreadPoolExecutor;

/**
 * 
 */
@Configuration
public class RedissonConfig {

    @Value("${redis.url}")
    private String redisUrl;

    @Value("${redis.poolsize}")
    private int poolSize;

    @Bean(value = "redisson")
    public Redisson getRedisson() {
        Config config = new Config();
        Redisson redisson = null;
        try {
            config.useSingleServer()
                    .setAddress(new StringBuilder().append("redis://").append(redisUrl).toString())
                    .setFailedAttempts(0) // 设置失败重试次数
                    .setConnectTimeout(100) // 设置链接超时时间
                    .setRetryAttempts(1) // 设置重试次数 
                    .setTimeout(100); // 设置超时时间

            redisson = (Redisson) Redisson.create(config);
            return redisson;
        } catch (Exception e) {
            log.error("getRedisson error {}", e);
        }
    }
    @Bean
    public ThreadPoolTaskExecutor redisCacheExecutor() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        // 设置核心线程数
        executor.setCorePoolSize(2);
        // 设置最大线程数
        executor.setMaxPoolSize(4);
        // 设置队列容量
        executor.setQueueCapacity(100);
        // 设置线程活跃时间（秒）
        executor.setKeepAliveSeconds(200);
        // 设置默认线程名称
        executor.setThreadNamePrefix("RedissonPlus-");
        // 设置拒绝策略
        executor.setRejectedExecutionHandler(new ThreadPoolExecutor.AbortPolicy());
        // 等待所有任务结束后再关闭线程池
        executor.setWaitForTasksToCompleteOnShutdown(true);
        return executor;
    }

}
