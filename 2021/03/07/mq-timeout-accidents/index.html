那些年在商城奋斗的光辉岁月 - MQ超时导致的事故

发表于 2021-03-07 更新于 2023-03-12 分类于 stability 阅读次数： 阅读次数： Valine：
本文字数： 4.5k 阅读时长 ≈ 4 分钟

事故简述：
MQ更换机器，替换过程中出现问题
支付流程中依赖MQ，发MQ超时导致支付-预付单接口超时。

(1) 事故描述

背景
MQ某共享集群部分机器过保(过了保修期)，集群全部nameserver节点均过保，替换过程中，部分业务出现生产消费流量突降；
支付调用MQ sdk向MQ写消息超时(共耗时8秒)，虽然MQ不是强依赖，但导致了上游发起支付超时，最终支付失败;


(1.1) 事故时间线

02-24 16:02 MQ共享集群部分机器过保，研发用新机器替换已经过保的nameserver机器；
02-24 16:02 MQ开始发生故障 事故开始
02-24 16:03 MQ研发操作同学观察流量，发现消费侧流量有下降；
02-24 16:03 优选-微信小程序发起支付失败
02-24 16:04 MQ研发回滚Proxy模块配置，Proxy中添加nameserver地址。流量未恢复，开始登陆机器排查；（事后确定是导致生产失败的原因）

02-24 16:04 MQ消防群有业务一级报警、支付收银监控图上曲线异常
02-24 16:05 支付研发开始排查，收银请求支付createPayOrder接口报错增多；支付收到错误日志报警
02-24 16:07 支付研发发现报错原因是proxy模块发MQ消息失败，因为该消息是弱依赖，未继续排查其他MQ消息失败的原因，通知组内同学共同排查。
02-24 16:07 MQ研发根据日志分析，路由信息没有拿到，但ns上没有问题，启动重启proxy操作，操作节点流量有上升。
02-24 16:07 发现生产侧流量急剧下降。 事故严重
02-24 16:07 支付开始收到多个一级报警。排查问题，现象是支付-收银请求支付下单超时。支付-收银有上线，操作回滚（事后结论：与本次事故无关）
02-24 16:10 支付监控发起成功率正常，回调下跌，且多支付渠道均下跌，联系sre排查回调网络问题。
02-24 16:13 支付研发查看日志发现支付createPayOrder接口发MQ消息超时过大(3S)会导致接口整体超时，影响支付。 发现原因
02-24 16:16 找到降级开关，并开始操作降级。
02-24 16:16 MQ研发定位到broker上报的问题，在broker中启用老的ns地址，流量全部恢复； 恢复
02-24 16:16 监控曲线已恢复(MQ回滚)
02-24 16:17 完全恢复

(3.1) 问题直接原因

  1. MQ在更换nameserver机器时，client端(producer)的nameserver地址(ip:port)没有生效， client -> broker 异常
  2. 收银支付服务创建收银支付订单时发MQ异常，导致预支付接口超时。
  3. 业务方调用预收银支付接口(prepay)失败，导致支付流程异常，用户无法支付。

(3.2) 问题原因

(3.2.1) 为什么MQ的client端nameserver地址没有生效？

MQ服务 client、broker代码没变，以前也有类似更换，以前怎么没问题？

(3.2.1.1) MQ更换nameserver流程

本次过保机器替换，主要是存储元数据的nameserver，涉及broker和proxy模块，broker(存储模块)向nameserver上报涉及自己的元数据，proxy向nameserver请求，获取元数据信息，具体替换流程如下：


1.扩容新节点，并把新扩容的节点通知给broker，让其上报元数据；
2.待新老节点元数据一致后，将新节点下发给proxy，proxy内部重启开源版本的client，更新nameserver地址;
3.同时更新给broker的nameserver地址，去除老节点，只保留新节点；

回滚失效
在16:04的回滚流程中，正常应该是上面流程中的逆流程，在实际操作中，遗漏第三步，会导致proxy去请求没有元数据的nameserver，依旧获取不到元数据，如下图，导致流量没有恢复，并对后续排查方向有一定的误导；


(3.2.1.2) 消费下降原因

下降主要原因是在第二部，新的nameserver地址下发给proxy，proxy内部的rocketmq client也进行了重启，但是新的nameserver地址没有生效；

https://github.com/apache/rocketmq/blob/main/client/src/main/java/org/apache/rocketmq/client/impl/MQClientManager.java#L47

public MQClientInstance getOrCreateMQClientInstance(final ClientConfig clientConfig, RPCHook rpcHook) {
    // rocketmq client做了资源复用，部分client实例，在与broker交互部分有复用
    // clientId中没有考虑nameserver变化场景，生成的clientid依旧一致
    String clientId = clientConfig.buildMQClientId();
    MQClientInstance instance = this.factoryTable.get(clientId);
    if (null == instance) {
        instance =
            new MQClientInstance(clientConfig.cloneClientConfig(),
                this.factoryIndexGenerator.getAndIncrement(), clientId, rpcHook);
        MQClientInstance prev = this.factoryTable.putIfAbsent(clientId, instance);
        if (prev != null) {
            instance = prev;
            log.warn("Returned Previous MQClientInstance for clientId:[{}]", clientId);
        } else {
            log.info("Created new MQClientInstance for clientId:[{}]", clientId);
        }
    }

    return instance;
}

rocketmq client做了资源复用，部分client实例，在与broker交互部分有复用，如上图，clientId中没有考虑nameserver变化场景，生成的clientid依旧一致，使得新创建的rocketmq client底层依赖没有更新，链接的nameserver依旧是之前的要下线的nameserver；之前操作没有全部替换，并且节点比较少，更新完立刻停服，proxy对nameserver的请求间隔是30s

(3.2.1.3) 生产下降原因

因为回滚了proxy上的ns配置，proxy连的ns上没有数据。

(3.2.1.4) 综合原因

  1. 变更proxy上的nameserver操作其实之前执行过多次，均未出现问题。区别是以前都是增减一台的操作，不会出现问题，而这次的操作是全量的替换，测试和历史操作都未覆盖到这个case。 (操作的问题还是预案的问题？)
  2. 变更发现问题后回滚未及时止损。

(3.2.2) 为什么收银支付服务的预支付接口超时？

以前怎么不超时？

当前的发起支付链路是：支付(phoenix) → 收银(Cashier) (→ 网关proxy ， → Scheduler → MQ)

phoenix 支付服务(收银台)
Cashier 收银服务/收银支付
proxy 渠道网关

支付Cashier收到发起支付请求之后做两件事：
1.调用渠道网关向指定的三方支付渠道发起支付（如微信、支付宝等）
2.再调用Scheduler向MQ发送一条消息（本case耗时8秒严重超时），注册延时任务，目的是在一定时间后主动调用渠道查询支付状态。（弱依赖 可降级，如果没有的话，就靠用户手动查）

8秒 = 4次 * (每次 获取连接1s + 写MQ消息1s)

/carrera/producer/examples/CarreraProducerExample.java

CarreraConfig config = new CarreraConfig(Env.Test);
config.setCarreraProxyTimeout(50);
config.setCarreraClientTimeout(100);
config.setCarreraClientRetry(2);
config.setCarreraPoolSize(20);
config.setBatchSendThreadNumber(16);
CarreraProducer producer = new CarreraProducer(Arrays.asList("TOPIC0", "TOPIC1", "TOPIC2"), config);
//启动producer，不可忽略.
producer.start();

carreraProxyTimeout 默认值=50 Proxy端处理请求的超时时间。写队列超时之后会尝试Cache。Cache成功后会返回CACHE_OK
carreraClientTimeout 默认值=100 client和proxy server的超时时间，一般不建议设太小。必须大于carreraProxyTimeout的值，建议设置2倍的比例
carreraClientRetry 默认值=2 客户端失败重试次数
carreraPoolSize 默认值=20 实例池中启动的producer最大连接数
batchSendThreadNumber 默认值=16 消息并发发送的并发线程数，仅使用sendBatchConcurrently方法时需要配置

(3.2.3) 为什么业务方调用预收银支付接口(prepay)失败？

  1. 收银支付接口(prepay)超时，导致调用失败。
  2. 收银支付接口是强依赖，当时没有备用方案。
  3. 当时优选业务的支付方式里没有离线支付(小额免密支付、先用后付等)
