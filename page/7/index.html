(3) 问题分析

(3.1) 问题直接原因

  1. MQ在更换nameserver机器时，client端(producer)的nameserver地址(ip:port)没有生效， client -> broker 异常
  2. 收银支付服务创建收银支付订单时发MQ异常，导致预支付接口超时。
  3. 业务方调用预收银支付接口(prepay)失败，导致支付流程异常，用户无法支付。

(3.2) 问题原因

(3.2.1) 为什么MQ的client端nameserver地址没有生效？

MQ服务 client、broker代码没变，以前也有类似更换，以前怎么没问题？

(3.2.1.1) MQ更换nameserver流程

本次过保机器替换，主要是存储元数据的nameserver，涉及broker和proxy模块，broker(存储模块)向nameserver上报涉及自己的元数据，proxy向nameserver请求，获取元数据信息，具体替换流程如下：

1.扩容新节点，并把新扩容的节点通知给broker，让其上报元数据；
2.待新老节点元数据一致后，将新节点下发给proxy，proxy内部重启开源版本的client，更新nameserver地址;
3.同时更新给broker的nameserver地址，去除老节点，只保留新节点；

回滚失效
在16:04的回滚流程中，正常应该是上面流程中的逆流程，在实际操作中，遗漏第三步，会导致proxy去请求没有元数据的nameserver，依旧获取不到元数据，如下图，导致流量没有恢复，并对后续排查方向有一定的误导；

(3.2.1.2) 消费下降原因

下降主要原因是在第二部，新的nameserver地址下发给proxy，proxy内部的rocketmq client也进行了重启，但是新的nameserver地址没有生效；

(3.2.2) 为什么收银支付服务的预支付接口超时？

以前怎么不超时？

当前的发起支付链路是：支付(phoenix) → 收银(Cashier) (→ 网关proxy ， → Scheduler → MQ)

phoenix 支付服务(收银台)
Cashier 收银服务/收银支付
proxy 渠道网关

支付Cashier收到发起支付请求之后做两件事：
1.调用渠道网关向指定的三方支付渠道发起支付（如微信、支付宝等）
2.再调用Scheduler向MQ发送一条消息（本case耗时8秒严重超时），注册延时任务，目的是在一定时间后主动调用渠道查询支付状态。（弱依赖 可降级，如果没有的话，就靠用户手动查）

8秒 = 4次 * (每次 获取连接1s + 写MQ消息1s)

(3.2.3) 为什么业务方调用预收银支付接口(prepay)失败？

  1. 收银支付接口(prepay)超时，导致调用失败。

  3. 当时优选业务的支付方式里没有离线支付(小额免密支付、先用后付等)

使用Lua脚本的好处：

减少网络开销。可以将多个请求通过脚本的形式一次发送，减少网络时延。
原子操作。redis会将整个脚本作为一个整体执行，中间不会被其他命令插入。因此在编写脚本的过程中无需担心会出现竞态条件，无需使用事务。
复用。客户端发送的脚本会永久存在redis中，这样，其他客户端可以复用这一脚本而不需要使用代码完成相同的逻辑。

EVAL script numkeys key [key ...] arg [arg ...]

说明：
script是第一个参数，为Lua 5.1脚本。该脚本不需要定义Lua函数（也不应该）。
numkeys指定后续参数有几个key。
key [key ...]，是要操作的键，可以指定多个，在lua脚本中通过KEYS[1], KEYS[2]获取
arg [arg ...]，参数，在lua脚本中通过ARGV[1], ARGV[2]获取。

那些年我们在商城奋斗的光辉岁月 - redis超时(延迟高)导致的事故

发表于 2021-02-28 更新于 2023-02-18 分类于 stability 阅读次数： Valine：

2月26，元宵节当天，商城出了一个事故，😭，简单地说是一个 小bug + redission使用不当 + 高并发流量 + 强校验 导致的一个大问题。

(1) 问题时间线

02-25 20:48 新需求上预发验证时，遇到个小问题，代码里忘记判空，导致某个key缓存了全表数据(全量活动标签)，随后redis集群里的一个分片 CPU飙高开始报警；
02-25 21:00 预发代码回滚完成。
　　但是缓存没有清理。而且线上和预发用的是同一个redis集群。(给事故埋下伏笔)
　　由于晚上流量逐渐变小，所以线上未出现问题。

02-26 08:32 活动服务的redis查询延时升高告警
02-26 08:35 ～ 09:02 redis连接延时升高、导致查询活动信息延时升高，下单调活动超时，触发报警 (事故开始)

02-26 09:02 由于天降惊喜活动相关查询减少，慢请求减少，redis QPS下降，redis proxy恢复正常，活动接口耗时下降，下单服务调活动接口恢复正常 (事故结束)
02-26 13:05 删除问题缓存，代码修复发布上线，redis查询QPS趋于稳定，完全恢复

8:48 ~ 8:55是查询最慢的时间 监控


(2) 问题分析

(2.1) 事故直接原因

  1. redis某个分片查询耗时高(最高有200ms)
  2. redis proxy查询耗时高(300-500ms)，导致活动接口耗时高
  3. 活动接口耗时高(>500ms) 导致 下单调活动超时
  4. 下单强依赖活动，调活动失败后返回网络错误

(2.2) 问题原因

请求路径 redis client –> VIP:PORT –> 均摊到16个proxy机器 –> 热点key+大key落到的某一台redis分片

由于请求在proxy堆积，导致所有的proxy的cpu都飙升了。

备注 redis查询耗时 2~200ms，而redis proxy耗时有 300 ~ 500 ms

(2.2.1) 为什么redis有一个分片查询耗时高？

以前redis所有分片的查询都在1~5ms，而事故发生时redis有1个分片耗时最高有200ms，为什么？

  1. 事故发生前一个有个bug，导致出问题的redis分配里某个key是大key。 (key对应的value是全表的数据)
  2. redis出问题那个分片上的查询是个复杂lua脚本，会操作hash zset，而且会对hash里的所有field操作，特别耗费性能。
  3. 代码里有循环调redis的情况，正好查的是那个分片
  4. 调redis超时后redission默认重试3次，导致流量翻倍
  5. 出问题那个分片上的流量还比较大，相当于热key

大key + 热key + 复杂lua脚本 + 流量翻倍 导致有1个redis分片耗时高(200ms)

(2.2.2) 为什么所有redis proxy查询耗时高?

以前redis proxy查询<1ms，而事故发生时16个redis proxy耗时都有 300 ~ 500 ms ，为什么?

  1. (根因) Redis有1个分片的 热key + 大key + 复杂lua脚本 导致redis的请求耗时高。
  2. (直接原因) 请求Redis耗时高，并且流量大，而请求均匀分布在16个redis proxy，请求堆积，导致所有redis proxy耗时高。
  3. 调用redis失败后又新建连接，导致redis proxy连接数增加，性能下降，延迟增加。
  4. 代码里有循环调redis的情况，导致redis proxy CPU压力较大
  5. 调redis-proxy超时后redission默认重试3次，导致流量翻倍 (1~3倍)
  6. 用户有重试，导致流量翻倍 (2~10倍)

redis一个分片耗时高 + 请求堆积 + 连接数增多 + 流量翻倍 综合导致redis proxy CPU飙升，请求堆积，所有请求都耗时。

proxy是请求堆积才引起的CPU利用率升高，
根因还是有一个Redis分片上请求变慢，Redis变慢的原因是这个分片上有lua脚本访问热key，请求量比别的分片高

相同qps，连接数增加，延迟也会大幅增加


(2.2.3) 为什么活动查询接口耗时高？

以前活动接口查询耗时在30~200ms，为什么事故发生时耗时超过500ms ?

  1. 调redis获取数据耗时高，有300 ~ 500ms。
  2. 代码里有循环调redis的情况，导致耗时翻倍。
  3. 活动查询标签的代码里有默认查询新用户的逻辑(这个场景不应该走这段代码)，也比较耗时。
  4. 查询活动接口会判断这个商品是否是秒杀、新人、万人团、促销，一共会调用4次redis，耗时较高
  5. 活动服务里redis的默认超时时间是3秒
  6. 调redis没有熔断机制。
  7. 调redis超时后，又创建tcp连接请求redis，共有 64个tcp连接/台 * 200台机器 = 12800个tcp连接
  8. 活动服务web容器(Tomcat)连接数被打满。
  9. 调用方流量变大(用户重试)，导致流量放大。

备注：活动服务redis配置了每台机器最多64个连接

(2.2.4) 为什么下单失败了？

  2. 下单对活动是强依赖 (历史原因 下单要拿到活动信息给秒杀商品超卖兜底)。

(2.3) 问题lua脚本

Redis提供的hash只有key有过期时间，而hash对象的 field value上是没有过期时间的。
这个lua脚本的作用是给 hash结构里的 field value 设置一个过期时间。

下面命令比较复杂，简化一下就是 EVAL lua_script 5 key1 key2 key3 key4 key5 value1 value2
script是第一个参数，为Lua脚本。
5代表后续参数有5个key。
key1 key2 key3 key4 key5，是要操作的键。
value1 value2，代表有2个参数，在lua脚本中通过ARGV[1], ARGV[2]获取。

下面是根据redis slowlog拿到的一个样例

-- ARGV[1] 是入参里的value1 对应 过期时间    "1614301199559"

--

local value = redis.call('hget', KEYS[1], ARGV[2]);
-- hget key field 对应的值为空
if value == false then
    return nil;
end ;

-- 将二进制反序列化，解析数据
local t, val = struct.unpack('dLc0', value);
-- 默认过期时间 无限大
local expireDate = 92233720368547758;
-- 从zset里获取field对应的过期时间   zscore key member
local expireDateScore = redis.call('zscore', KEYS[2], ARGV[2]);
-- 过期时间 不等于 false
if expireDateScore ~= false then
    -- 解析过期时间
    expireDate = tonumber(expireDateScore)
end ;

-- t不等于0
if t ~= 0 then
    local expireIdle = redis.call('zscore', KEYS[3], ARGV[2]);
    -- expireIdle 不等于 false
    if expireIdle ~= false then
        if tonumber(expireIdle) > tonumber(ARGV[1]) then
            -- 将二进制反序列化，
            local value = struct.pack('dLc0', t, string.len(val), val);
            -- 更新 hash对象 key field对应的值
            redis.call('hset', KEYS[1], ARGV[2], value);
            -- 更新zset对象 key 的值
            redis.call('zadd', KEYS[3], t + tonumber(ARGV[1]), ARGV[2]);
        end ;
        -- 过期时间
        expireDate = math.min(expireDate, tonumber(expireIdle))
    end ;
end ;

-- 如果 过期时间 <= 入参里的value1过期时间
if expireDate <= tonumber(ARGV[1]) then
    return nil;
end ;

-- hget key5 'max-size'
local maxSize = tonumber(redis.call('hget', KEYS[5], 'max-size'));
-- 如果 max-size 对饮的值为空 并且 不等于0
if maxSize ~= nil and maxSize ~= 0 then
    -- zadd key4 score(对应入参value1的时间) member(对应hash的field)
    redis.call('zadd', KEYS[4], tonumber(ARGV[1]), ARGV[2]);
end ;

-- 返回值

lua脚本给hash key field设置过期时间原理

从上面的lua脚本可以看出 redisson给hash对象每个field设置过期时间的原理

  1. 对hash对象的key和field 对应设置过期时间
    zadd key score(过期时间) member(对应hash对象的field)

  2. 在hget key field时获取过期时间
    通过 zscore key member(对应hash对象的field) 获取分数值(score)，这个score就是过期时间

下面是用redis命令实现上面lua脚本的功能
hset key1 field1 value1
zadd key4(≈key1) score(过期时间) member(对应hset里的field1)
zscore key3(≈key1) member(对应haset里的field1) 获取的值就是过期时间

key4和key1是关联的， key4 = "redisson__map_cache__last_access__set:{" + key1 + "}"
key3和key1是关联的， key3 = "redisson__idle__set:{ + key1 + }"

    if(CollectionUtils.isEmpty(goodsIdList)){
        return Collections.emptyMap();
    }

    try {
        // key是 goods_label:cityId
        RMapCache<Long, List<ActivityGoodsLabel>> mapCache = redissonClient.getMapCache(buildCacheKey(cityId));

        List<Long> needQueryList = Lists.newArrayList();
        Map<Long, List<ActivityGoodsLabel>> resultGoodsLabelMap = Maps.newHashMap();

        if (MapUtils.isNotEmpty(mapCache)) {
            // 循环里调mapCache.get() 踩坑了 
            goodsIdList.forEach(goodsId -> {  // 按照goodsId逐个设置label
                if (mapCache.containsKey(goodsId)) {
                    // 缓存查到部分数据  // 这块mapCache.get(goodsId) 相当于hget goodsId 
                    resultGoodsLabelMap.put(goodsId, mapCache.get(goodsId));
                } else {
                    //未查到的数据
                    needQueryList.add(goodsId);
                }
            });

            if (CollectionUtils.isEmpty(needQueryList)) {
                return resultGoodsLabelMap;
            }
        }else {
            // 需要从数据库查的goodsIdList
            needQueryList.addAll(goodsIdList);
        }
        // 从数据库查询goodsIds对应的标签
        Map<Long, List<ActivityGoodsLabel>> goodsLabelMap = doQueryLabel(typeList, needQueryList);

        // 把从数据库查到的数据添加到缓存
        // 循环调用redis 
        goodsLabelMap.forEach((key, val) -> mapCache.put(key, val, 5, TimeUnit.SECONDS));

        resultGoodsLabelMap.putAll(goodsLabelMap);
        //返回值设置
        return resultGoodsLabelMap;

    }catch (Exception e){
        log.warn("查询label出现异常，err:{}", e.getMessage(), e);
        return doQueryLabel(typeList, goodsIdList);
    }
}

(3) 问题分析

  1. 下单时触发支付限流，创建订单成功，创建收银支付订单失败，下单报错，对用户提示“调用收银台返回异常”
  2. 用户继续点击，触发限购，对用户提示“秒杀商品xx，您的购买已经超过上限”
  3. 用户继续点击，对用户提示“请勿重复提交”

下单支付流程

(3.1) 问题直接原因

  1. 商品秒杀活动瞬时流量 + 用户活动流量叠加，整体流量较高。
  2. 下单时收银支付接口限流导致下单失败(创建订单成功-生成预付单失败)。
  3. 前端交互存在问题，服务端下单接口返回错误码，没有跳转到对应页面。
  4. 前端在用户点击下单支付按钮后，没有把按钮置灰。
  5. 用户点击立即支付按钮后没有成功，重复点击，导致流量翻倍。

(3.2) 问题详细原因

(3.2.1) 为什么下单流量高了？

  1. 优选业务处在快速增长时期，一直在开城，流量每天增高是正常现象。
  2. 问题发生事件在年底(腊月初八 ~ 腊月十二)，用户在备年货，运营在搞活动，团队在冲业绩，商品活动和用户活动较多。
  3. 秒杀商品价格低于成本价，导致购买用户较多，瞬时交易流量较大。
  4. 下单失败用户重试导致流量翻倍。

(3.2.2) 为什么收银支付触发限流？

  1. 下单流程里包含创建订单和创建收银支付单。
  2. 下单的QPS高了导致收银支付的QPS也高了。
  3. 当时只接了收银支付的一个机房，QPS达到收银支付那个机房的阈值，触发限流。

(3.2.3) 为什么前端交互存在问题？

按照预期的流程，用户点击立即支付按钮后，前端调服务端下单接口，下单创建订单成功，生成预付单失败，返回了错误码，前端根据错误码跳转到订单列表提示用户二次支付；
但实际没有跳转。

  1. 历史原因，前端不知道有个错误码没有处理。
  2. 其它错误码都处理了，也可能是前端搞出的bug。 (前端合代码冲突 或 误删代码)
  3. 在微信小程序交易场景，没有收银台，默认支付方式微信支付，前端调完下单接口即调用微信支付SDK，这样用户体验好，交易效率也高。

(3.2.4) 为什么用户会重复点击？

  1. 秒杀活动商品价格低于成本价，用户都想抢，迟了可能会售罄。
  2. 服务端下单接口返回错误码和错误信息后，前端处理成提示了。
  3. 前端没有把立即支付按钮置灰，用户可以一直点击。
  4. 服务端下单接口幂等时间较短。

package com.alibaba.csp.sentinel.adapter.dubbo;

/**
 * 支持与Sentinel集成的Apache Dubbo服务提供商筛选器。默认情况下自动激活。
 * 注意：这只适用于Apache Dubbo 2.7.x或更高版本。
 *  
 * 如果要禁用提供程序筛选器，可以配置： dubbo:provider filter="-sentinel.dubbo.provider.filter" 
 */
@Activate(group = PROVIDER)
public class SentinelDubboProviderFilter extends BaseSentinelDubboFilter {

    public SentinelDubboProviderFilter() {
        RecordLog.info("Sentinel Apache Dubbo provider filter initialized");
    }

    @Override
    String getMethodName(Invoker invoker, Invocation invocation, String prefix) {
        return DubboUtils.getMethodResourceName(invoker, invocation, prefix);
    }

    @Override
    String getInterfaceName(Invoker invoker, String prefix) {
        return DubboUtils.getInterfaceName(invoker, prefix);
    }

    @Override
    public Result invoke(Invoker<?> invoker, Invocation invocation) throws RpcException {
        // Get origin caller.
        String origin = DubboAdapterGlobalConfig.getOriginParser().parse(invoker, invocation);
        if (null == origin) {
            origin = "";
        }
        Entry interfaceEntry = null;
        Entry methodEntry = null;
        String prefix = DubboAdapterGlobalConfig.getDubboProviderResNamePrefixKey();
        // 接口名
        String interfaceResourceName = getInterfaceName(invoker, prefix);
        // 方法名 
        String methodResourceName = getMethodName(invoker, invocation, prefix);
        try {
            // Only need to create entrance context at provider side, as context will take effect
            // at entrance of invocation chain only (for inbound traffic).
            // 使用方法名作为上下文名称
            ContextUtil.enter(methodResourceName, origin);
            // 使用接口名作为interfaceEntry资源名
            interfaceEntry = SphU.entry(interfaceResourceName, ResourceTypeConstants.COMMON_RPC, EntryType.IN);
            // 使用方法名作为methodEntry资源名
            methodEntry = SphU.entry(methodResourceName, ResourceTypeConstants.COMMON_RPC, EntryType.IN,
                invocation.getArguments());

            // 调用方法    
            Result result = invoker.invoke(invocation);
            if (result.hasException()) {
                Tracer.traceEntry(result.getException(), interfaceEntry);
                Tracer.traceEntry(result.getException(), methodEntry);
            }
            return result;
        } catch (BlockException e) {
            // 触发限流或熔断 
            return DubboAdapterGlobalConfig.getProviderFallback().handle(invoker, invocation, e);
        } catch (RpcException e) {
            Tracer.traceEntry(e, interfaceEntry);
            Tracer.traceEntry(e, methodEntry);
            throw e;
        } finally {
            if (methodEntry != null) {
                // 退出
                methodEntry.exit(1, invocation.getArguments());
            }
            if (interfaceEntry != null) {
                // 退出
                interfaceEntry.exit();
            }
            ContextUtil.exit();
        }
    }

}

package com.alibaba.csp.sentinel.annotation.aspectj;

/**
 * SentinelResource注解切面
 */
@Aspect
public class SentinelResourceAspect extends AbstractSentinelAspectSupport {

    /** 指定切入点为@SentinelResource注解 */
    @Pointcut("@annotation(com.alibaba.csp.sentinel.annotation.SentinelResource)")
    public void sentinelResourceAnnotationPointcut() {
    }

    /** 指定为环绕通知 */
    @Around("sentinelResourceAnnotationPointcut()")
    public Object invokeResourceWithSentinel(ProceedingJoinPoint pjp) throws Throwable {
        Method originMethod = resolveMethod(pjp);

        SentinelResource annotation = originMethod.getAnnotation(SentinelResource.class);
        if (annotation == null) {
            // Should not go through here.
            throw new IllegalStateException("Wrong state for SentinelResource annotation");
        }
        // 资源名 使用注解里配置的资源名或者方法名
        String resourceName = getResourceName(annotation.value(), originMethod);
        // 注解里没配置 默认是OUT
        EntryType entryType = annotation.entryType();
        // 默认为0
        int resourceType = annotation.resourceType();
        Entry entry = null;
        try {
            // 进行流控
            entry = SphU.entry(resourceName, resourceType, entryType, pjp.getArgs());
            // 通过流控检查后，才会调用目标方法
            return pjp.proceed();
        } catch (BlockException ex) {
            // 对BlockException进行回调
            return handleBlockException(pjp, annotation, ex);
        } catch (Throwable ex) {
            Class<? extends Throwable>[] exceptionsToIgnore = annotation.exceptionsToIgnore();
            // The ignore list will be checked first.
            if (exceptionsToIgnore.length > 0 && exceptionBelongsTo(ex, exceptionsToIgnore)) {
                throw ex;
            }
            if (exceptionBelongsTo(ex, annotation.exceptionsToTrace())) {
                traceException(ex);
                // 异常回调
                return handleFallback(pjp, annotation, ex);
            }

            // No fallback function can handle the exception, so throw it out.
            throw ex;
        } finally {
            if (entry != null) {
                entry.exit(1, pjp.getArgs());
            }
        }
    }

}

进入SentinelResource切面后，会执行SphU.entry()方法，在这个方法中会对被拦截方法做限流和熔断的逻辑处理。
如果触发限流或熔断，会抛出BlockException，我们可以指定blockHandler方法来处理BlockException。而对于业务上的异常，我们也可以配置fallback方法来处理被拦截方法调用产生的异常。

package com.alibaba.csp.sentinel.slots.clusterbuilder;
 
/**
 * 该槽维护资源运行统计信息（响应时间、qps、线程数、异常），以及调用者列表，由 ContextUtil#enter(String origin)标记 
 * 一个资源只有一个集群节点，而一个资源可以有多个默认节点。
 */
@Spi(isSingleton = false, order = Constants.ORDER_CLUSTER_BUILDER_SLOT)
public class ClusterBuilderSlot extends AbstractLinkedProcessorSlot<DefaultNode> {

    /**
     * 相同的资源 ResourceWrapper#equals(Object) 将在全局范围内共享相同的ProcessorSlotChain，无论在哪个上下文中。
     * 因此，如果代码进入 entry()，资源名称必须相同，但上下文名称可能不同。
     * 
     * 为了获取同一资源在不同上下文中的总统计信息，同一资源在全局共享相同的ClusterNode。所有ClusterNode都缓存在此映射中。
     * 
     * 应用程序运行的时间越长，这个映射就会变得越稳定。所以我们不是并发映射而是锁。因为这个锁只发生在最开始，而并发映射将一直持有锁。
     * 
     */
    private static volatile Map<ResourceWrapper, ClusterNode> clusterNodeMap = new HashMap<>();

    private static final Object lock = new Object();

    private volatile ClusterNode clusterNode = null;

    @Override
    public void entry(Context context, ResourceWrapper resourceWrapper, DefaultNode node, int count,
                      boolean prioritized, Object... args)
        throws Throwable {
        // 单例-DCL
        if (clusterNode == null) {
            synchronized (lock) {
                if (clusterNode == null) {
                    // 创建集群节点​​。
                    clusterNode = new ClusterNode(resourceWrapper.getName(), resourceWrapper.getResourceType());
                    HashMap<ResourceWrapper, ClusterNode> newMap = new HashMap<>(Math.max(clusterNodeMap.size(), 16));
                    newMap.putAll(clusterNodeMap);
                    // 添加到缓存里
                    newMap.put(node.getId(), clusterNode);
                    // 更新缓存
                    clusterNodeMap = newMap;
                }
            }
        }
        node.setClusterNode(clusterNode);

        /*
         * 如果设置了原始上下文，我们应该获取 或 创建原始直接的Node。
         */
        if (!"".equals(context.getOrigin())) {
            Node originNode = node.getClusterNode().getOrCreateOriginNode(context.getOrigin());
            context.getCurEntry().setOriginNode(originNode);
        }

        fireEntry(context, resourceWrapper, node, count, prioritized, args);
    }

}
